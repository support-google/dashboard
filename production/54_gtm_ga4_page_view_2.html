Page view Trigger
URL 기본 구조에 대한 이해

Domain / Path / Parameter / Fragment

    URL 구성요소 - Domain
    Domain   Page path Variable
    정의 : 도메인은 네트워크상의 컴퓨터 주소를 의미합니다. 도메인 전체에는 국가나 기관의 정보, 그리고 도메인이름과 도메인에 속한 호스트 정보가 기입됩니다. 각 정보는 점(.)으로 구분할 수 있으며, 뒤에서부터 순차적으로 확인할 수 있습니다.
    구조 : 도메인은 상위도메인, 도메인이름, 호스트명으로 이루어짐

        상위 도메인 명 : 도메인의 목적, 종류, 국가를 나타냄
        도메인 명 : 임의로 지정할 수 있는 사이트의 이름
        호스트 명 : 도메인에 속한 컴퓨터 이름, 도메인의 하위 개념

  • 'music.naver.com' 의 상세 도메인 구분 예시
    예시1 : music.naver.com/listen/top100.nhn에서, 도메인은 ‘music.naver.com’

        상위 도메인 명 : 'music.naver.com' 도메인에서, 영리 목적의 기업을 의미하는 'com'
        도메인 명 : 'music.naver.com' 도메인에서, 임의로 지정한 도메인명은 'naver'
        호스트 명 : 'music.naver.com' 도메인에서, 호스트명은 naver.com라는 도메인에 속한 'music'

    URL 구성요소 - Path
    Path(/)   Page path Variable
    정의 : 파일 도메인에 해당하는 웹 서버를 기준으로, 제공하고자 하는 페이지가 속한 폴더명과 파일명이 나타난 경로를 나타냄
    '/'(슬래시) 로 구분되는 형태이며 컴퓨터의 폴더와 비슷한 개념
    구조 : 파일 경로는 도메인 뒤 '/'부분에 해당하며, 파일 경로와 파일 명으로 이루어짐

        파일 경로 : 도메인 웹서버내 하위 경로(폴더)
        파일 명 : 하위 경로(폴더)내 저장되어 있는 파일

  • '/listen/top100.nhn' 의 상세 경로 구분 예시
    예시1 : music.naver.com/listen/top100.nhn에서, 파일 경로는 '/listen/top100.nhn'

        파일 경로 : 'music.naver.com' 라는 도메인에 해당하는 웹 서버의 'listen' 폴더
        파일 명 : 'listen'라는 폴더내 저장되어 있는 'top100.nhn'파일

    URL 구성요소 - Parameter
    Parameter(Query String, ?)   Page path Variable
    정의 : 파라미터는 쿼리 스트링이라고도 하며, 정보에 따라서 페이지의 콘텐츠가 가변적일 수 있을 때 사용하거나 한 페이지 url을 상황에 따라 구분하기 위한 추가정보의 용도로 사용할 수 있음
    구조 : key(파라미터의 이름)=value(파라미터의 값) 형태로 구성됨. 파라미터는 url 의 파일명 뒤 물음표 이후에 나열이 되며, 이름과 값으로 구성된 세트가 1개 이상 존재할 수 있음. 이 세트는 엔드(&) 기호로 구분

        Key : 파라미터를 통해 실제로 수집하고자 하는 항목
        Value : 파라미터 항목에대한 실질적인 값

  • 파라미터를 한 페이지 url을 상황에 따라 구분하기 위한 추가정보의 용도로 사용하는 경우
    예시1 : https://music.naver.com/listen/top100.nhn/?utm_source=google 파라미터는 '?' 뒤 utm_source=google

        Key : 'utm_source=google'에서 소스 정보를 구분하기 위한 파라미터 이름 'utm_source'
        Value : 'utm_source=google'에서 실질적인 소스 값인 'google'

    의미 : 유저가 유입된 소스 정보는 google이라는 의미. UTM(Urchin Tracking Module)은 query parameter를 이용한 사용자 추적 서비스를 제공함
  • 페이지는 하나이지만 파라미터 옵션에 따라 다른 콘텐츠나 레이아웃으로 구성하는 경우
    예시2 : http://music.naver.com/listen/top100.nhn?domain=OVERSEA&duration=1h#content 의 파라미터는
    '?' 뒤 'domain=OVERSEA&duration=1h#content'

        Key : 'domain'과 'duaration'는 파라미터 이름. '지역'과 '기간'이라는 옵션으로 페이지 콘텐츠가 가변적으로 변경 됨
        Value : 'OVERSEA'와 '1h'는 파라미터 값. 각각의 파라미터에 옵션 값을 설정하여 제공함

    의미 : '~top100.nhn' 페이지를 보여줄때, domain과 duration라는 파라미터의 값을 각각 OVERSEA, 1h로 설정하여 제공
    이는 지역과 기간 옵션에 따라 페이지가 가변적으로 변함을 유추할 수 있음
    URL 구성요소 - Fragment
    Fragment(Anchor, #)   Page path Variable
    정의 : 페이지의 특정 요소를 지시하는 기능을 가짐. 해시태그로 이동을 원하는 요소의 id를 링크로 연결하면, 스크롤 이동없이 바로 해당 위치로 이동
    이러한 기능은, 여러 콘텐츠가 나열 된 페이지의 특정 위치 정보를 사용자에게 함께 전달하고자 할 때 유용하게 쓰임
    구조 : URL의 Fragment는 Domain, Path, Parameter가 모두 나온 후 등장하는 '#'부분에 해당

        id : 콘텐츠를 이루는 요소 html 태그는 id라는 고유한 속성값을 가질 수 있음. 이동을 원하는 컨텐츠에 id 속성값을 지정
        # : id선택자로써 #이 사용됨. 지정한 id값에다 해시태그를 붙여 url 뒤에 적용하면, 네트워크는 id에 해당하는 컨텐츠를 식별하고 이동함

  • Fragment를 활용해 원하는 요소로 이동하는 경우 - content 영역
    예시1 : http://music.naver.com/listen/top100.nhn?domain=OVERSEA&duration=1h#content 에서, Fragment는 '#content'

        이동 위치 : top100.nhn 이라는 순위 자료중에서, content 영역을 지시하고 싶을 때 사용

  • Fragment를 활용해 원하는 요소로 이동하는 경우 - Portfolio 영역
    예시2 : http://www.kinoz.com/#portfolio 에서, Fragment는 '#portfolio'

        이동 위치 : 한 개의 긴 페이지에 여러 메뉴의 콘텐츠가 구성된 사이트에서, portfolio 메뉴에 해당하는 영역으로 이동시키고자 할 때 사용

Page View Variable

Page view Variable
4가지 Page view 변수 안내

Page Path / Page Hostname / Page URL / Page Referrer

    페이지뷰 변수 종류 1
    페이지 경로(Page Path) 변수   Page path Variable
  • 예시 URL
  • Page Path
  • https://music.naver.com/listen/top100.nhn/?domain=OVERSEA&duration=1h#content
  • /listen/top100.nhn/
    변수 기능 : 해당 변수는 도메인 이름(music.naver.com)과 파라미터(?domain=OVERSEA&duration=1h#content)사이의
    페이지 경로(/listen/top100.nhn)를 추출하는 기능을 함
    주요 이점 : 일반 웹사이트의 특정 페이지나 페이지 그룹을 타겟팅할 때 유용
    예시 : 아래의 예시는 페이지 경로가 "/listen/top100.nhn" 와 동일하다는 추가 조건으로 페이지 보기 트리거를 구현한 것
    해당 트리거는 URL에 존재하는 동적 파라미터 값에 관계없이 페이지 경로가 조건에 정확히 일치하는 경우에만 실행됨
    페이지뷰 변수 종류 2
    페이지 호스트 이름(Page Hostname) 변수   Page Host Variable
  • 예시 URL
  • Page Hostname
  • https://music.naver.com/listen/top100.nhn/?domain=OVERSEA&duration=1h#content
  • music.naver.com
    변수 기능 : 이 변수는 트리거 또는 다른 변수에서 사용할 수 있는 URL의 도메인 이름(hostname을 뜻함 여기서는 music.naver.com)을 추출하는 기능을 함
    주요 이점 : 여러 도메인을 추적하는 하나의 GTM 계정에서 특정 도메인에 대해 태그를 제한해야 하는 경우 유용하게 활용
    예시 : 아래의 예시는 페이지 도메인이"music.naver.com"이며, 경로가 "/listen/top100.nhn/" 일때 동작하도록 구현한 것
    특정 도메인에 대한 접근을 제한하기 위해 Page View 트리거에 도메인 조건을 변수로 추가 하였음
    페이지뷰 변수 종류 3
    페이지 URL(Page URL) 변수   Page URL Variable
  • 예시 URL
  • Page URL
  • https://music.naver.com/listen/top100.nhn/?domain=OVERSEA&duration=1h#content
  • https://music.naver.com/listen/top100.nhn/?domain=OVERSEA&duration=1h
    변수 기능 : 이 변수는 파라미터(?domain=OVERSEA&duration=1h)를 포함하여 페이지 URL에있는 모든 것을 반환하지만,
    Fragment(#content)부분을 제외시키는 기능을 함

    즉 페이지 URL에서 누락된 유일한 부분은 URL Fragment(#content)임을 확인할 수 있음

    주요 이점 : 페이지 경로(Path)와 파라미터(Parameters)가 조합된 주소를 대상으로 해야 할 때 유용
    예시 : 아래의 예시는 ?domain=domestic 이 포함된 특정 페이지에서만 태그가 실행되도록 트리거를 구현한 것
    특정 파라미터 값까지 조건으로 설정하려면 페이지 경로 대신 페이지 URL을 사용
    페이지뷰 변수 종류 4
    페이지 리퍼러 변수(Referrer)   Page Referrer Variable
  • 예시 URL
  • Page Referrer
  • https://music.naver.com/listen/top100.nhn/?domain=OVERSEA&duration=1h#content
  • https://www.google.com/
    변수 기능 : 이 변수는(가능한 경우) 최종적으로 내 웹사이트로 이동시킨 것으로 확인되는 참조 도메인을 보여주는 기능을 함

    이 데이터는 방문 페이지(Landing Page)에서만 사용할 수 있음. 내부 페이지간 이동 후 리퍼러 값을 확인한다면 그 값은 자신의 URL이 됨

    주요 이점 : 참조 웹사이트를 기반으로 특정 태그를 실행하려는 경우, 더 복잡한 트리거에서 이를 사용할 수 있음
    (예: 사용자가 Google 검색에서 유입된 경우, 웹페이지를 찾는데 소요된 사용자 정의 시간).
    예시 : 아래의 예시는 https://www.google.com/ 이 레퍼러로 표시되는 경우 태그가 실행되도록 트리거를 구현한 것
    웹사이트로 연결되는 Google 자연 검색 결과를 클릭하면 https://www.google.com/ 이 추천자로 표시될 가능성이 큼

Trigger Order

Page view Trigger
3가지 Page view 트리거의 작동방식

Pageview / DOM-ready / Window loaded

    트리거 작동 순서
    페이지뷰 트리거 종류 1
    페이지 뷰   Page view
    트리거 정의 : 페이지 뷰는 웹 브라우저가 해당 페이지를 로드하기 시작하는 시점에 바로 발동되는 트리거
    동작 순서 : 3개 옵션 중 첫 번째로 동작

    동작 시점 : 웹 브라우저가 해당 페이지를 로드하기 시작하는 순간으로, Head에 심은 GTM Container 스크립트가 실행되는 시점

    화면 : 아직 전체 페이지가 로드되지는 않아서 유저는 웹페이지에서 조회할 수 있는 내용이 없는 상태

    페이지뷰 트리거 종류 2
    DOM 사용 가능   DOM Ready
    트리거 정의 : HTML 콘텐츠가 로드되고 구문이 분석되었으며 페이지의 DOM 렌더링을 시작할 준비가 되었을 때 발동되는 트리거
    동작 순서 : 3개 옵션 중 두 번째로 동작

    동작 시점 : 브라우저가 HTML에서 전체 페이지의 구성을 완료하고 문서객체모델(DOM)을 파싱할 준비가 된 시점

    화면 : 유저가 웹페이지 일부 구성요소를 조회할 수 있음

    주의사항 : DOM과의 상호작용을 통해 변수를 입력하는 페이지뷰 기반 태그에서는 올바른 변수를 태그 관리자에서 사용할 수 있도록 이 트리거 유형을 사용해야 함

    페이지뷰 트리거 종류 3
    창 로드   Window Loaded
    트리거 정의 : 페이지에 필요한 모든 리소스가 전부 로드되었을 때 발동되는 트리거
    동작 순서 : 3개 옵션 중 세 번째로 동작

    동작 시점 : 자바스크립트 태그까지 페이지에 필요한 모든 리소스가 전부 로드된 시점

    화면 : 사용자는 제대로된 페이지의 UI를 보고 있을 가능성이 높음

  • DOM이 무엇인가요?

    문서 객체 모델(DOM; Document Object Model)은 XML, HTML 문서의 각 항목을 계층으로 표현하여 생성, 변형, 삭제할 수 있도록 돕는 인터페이스.

    쉽게 말해 HTML문서는 html, head, body로 body는 form, table, tr, td로 이루어지는 등 큰 오브젝트에서 작은 오브젝트로 체계적으로 이루어져 있다는 의미

    DOM 준비 상태에서는 브라우저는 모든 CSS 파일, 이미지, 자바스크립트 등의 경로만 알고 있음
Page view Trigger
Page view 트리거의 선택기준

Pageview / DOM-ready / Window loaded

  •     브라우저에서 조회하는 웹 페이지는 3번에 걸쳐서 로드 "페이지 뷰 > DOM 사용 가능 > 창로드" 순으로 로드
    각 페이지 뷰마다 사용할 수 있는 변수가 조금씩 달라지기 때문에 원하는 정보가 어느 페이지 뷰에서 로드 되는 지를 파악하고 트리거를 만들어야 함                 
    페이지 뷰 트리거 사용 시 이벤트 list에서 확인되는 변수
    이벤트 list에서 확인되는 페이지뷰 트리거 수집 변수   Page view Trigger

    창로드 기준 : Container Loaded 이벤트는 몇몇 variable value(데이터 영역 변수)가 undefined로 확인되지 않음

    창로드 기준 : Container Loaded는 아직 전체 페이지가 로드되지 않은 시점이기 때문에 웹페이지 구성 요소 (제목, 콘텐츠 id 등) 정보를 알 수가 없어서 변수를 채우지 못함.
    DOM 트리거 사용 시 이벤트 list에서 확인되는 변수
    이벤트 list에서 확인되는 DOM 트리거 수집 변수   DOM Trigger

    창로드 기준 : DOM Ready 이벤트는 variable value(데이터 영역 변수)가 모두 채워진 것을 확인할 수 있음

    창로드 기준 : HTML 태그가 로드된 DOM 시점에는 page.title이나 page.content_id를 불러오기 때문에 변수가 채워져있음
    따라서 page.title을 변수로 사용하고 싶다면 GTM Trigger 중 페이지뷰가 아닌 DOM Ready를 trigger로 선택해야 함

Process 1

태그 구성 단계

    01
    태그 구성 
     1번 : 태그 탭 선택
     2번 : 새로만들기를 클릭하면 오른쪽에서 태그 생성창이 열림
    태그 구성 
     3번 : 태그 이름 작성. 구글에서 지정한 공식 이벤트 이름인 'page_view'와 GA에서 구분될 수 있도록 별도의 이름으로 지정.
     4번 : [태그 구성]을 클릭하면 다시 한번 오른쪽에서 태그 목록이 열림
  • Process
    02
    태그 구성 
     1번 : Google 애널리틱스 : GA4 이벤트 선택
    태그 구성 
     2번 : 태그 유형에는 '[Google 애널리틱스 : GA4 이벤트]'가 제대로 선택됬는지 확인
     3번 : 구성 태그에는 '[GA4]측정 ID태그'를 선택(GA4의 측정ID를 불러오는 태그)
     4번 : 이벤트 이름에는 'page_view(part)'를 입력(소문자로만). 구글에서 지정한 공식 이벤트 이름인 'page_view'와 GA에서 구분될 수 있도록 별도의 이름으로 지정.

Process 2

2. 트리거 구성

트리거 구성 단계

  • Process
    03
    트리거 구성 
     1번 : [트리거]를 클릭하면 오른쪽에 트리거 유형을 선택할 수 있는 창이 열림
    트리거 구성 
     2번 : 트리거 선택에서 [+]를 클릭
  • Process
    04
    트리거 구성 
     1번 : 트리거 이름을 입력. '[GA4] particular page_view' 태그와 일관성을 유지하기 위해 트리거 이름은 '[GA4] particular page_view Trigger'로 입력함
     2번 : [트리거 구성]을 클릭한다
    트리거 구성 
     3번 : 트리거 유형에서 [페이지뷰]를 선택
  • Process
    05
    단일 트리거의 여러 URL 옵션1(사용 불가)  
     세팅 방법 : 하나의 트리거에 각 페이지 URL조건을 동시에 추가
    방문자가 3페이지 중 하나라도 방문하는 경우 태그를 실행시키는 트리거 옵션1
        불행히도 위와 같은 세팅은 트리거에서 모든 조건이 충족되어야 한다는 조건이기에(AND조건) 이 세팅은 작동하지 않음
        즉 하나의 페이지가 동시에 5개의 다른 URL을 가질 수 없기 때문에 해당 세팅은 잘못 됨
    단일 트리거의 여러 URL 옵션2(비권장)  
     세팅 방법 : 각각의 페이지에 대한 별도의 페이지뷰 기반 트리거를 생성한 후 모두 페이지뷰 태그에 추가
    방문자가 3페이지 중 하나라도 방문하는 경우 태그를 실행시키는 트리거 옵션2
        이러한 방법으로 세팅 시, 트리거 중 하나 이상이 활성화되면 태그가 실행됨
        그러나 이 솔루션은 확장성이 뛰어나지 않으며, GTM 컨테이너를 빠르게 엉망으로 만드는 방법임
  • Process
    06
    단일 트리거의 여러 URL 옵션3(권장)  
     세팅 방법 : 정규식을 사용하는 단일 트리거를 추가
         1번 : [페이지뷰]로 올바르게 세팅되었는지 확인
         2번 : 이 트리거는 [일부 페이지 뷰]에서만 실행되어야만 함
         3번 : [Page URL] 조건 선택
         4번 : 정규 표현식과 일치(대소문자 무시) 선택
         5번 : 정규식 /pages/contact-us|/pages/add_to_cart|/pages/pricing 입력
    방문자가 3페이지 중 하나라도 방문하는 경우 태그를 실행시키는 트리거 옵션3
        RegEx를 사용하면 "OR"을 포함한 더 복잡한 조건을 작성할 수 있음
        정규식에서 기호 "|"는 "OR"을 의미하며 2개의 URL을 대상으로 지정하려는 경우 해당 부분을 "|"로 분리할 수 있음
        물론 트리거가 작동하도록 하려면 정규 표현식과 일치(대소문자 무시)를 선택
    트리거 구성 
     7번 : 네모칸과 같이 세팅이 완료되었다면 정상적으로 진행 된 것
     8번 : 저장 클릭
  • Process
    07
    트리거 구성 
     1번 : 저장중 이라는 화면 뜸
    트리거 구성 
     2번 : 네모칸과 같이 세팅이 완료되었다면 정상적으로 진행 된 것

Process 4

4. 태그 최종 게시(웹사이트 배포)

작업 완료된 태그 최종 게시

  • Process
    05
    최종 게시 전 테스트 및 디버깅  
     조금 전 생성한 태그가 작업공간의 변경사항에 추가 된 것을 확인
     이상 없을 경우 미리보기 클릭
    최종 게시 전 테스트 및 디버깅

     [connect]를 클릭하면 홈페이지가 열리면서 미리보기 모드와 연결 됨
  • Process
    06
    최종 게시 전 테스트 및 디버깅  
     연결이 완료되면 팝업창이 Connected로 내용이 변경 됨
     Continuned를 클릭하여 테스트 실행
    최종 게시 전 테스트 및 디버깅  
     미리보기 화면에서 [Tags Fired] 영역에 설정해둔 태그들이 보인다면 제대로 작동하는 것
     의도한 대로 조건에 맞게 태그가 Fired 되는지도 체크
  • Process
    07
    태그 배포  
     실제로 웹 사이트에 가상 태그를 배포하는 작업

    태그 배포  
     1번 : 구글 태그 매니저는 협업으로 많이 활용하기에 해당 작업을 누가 했는지 표식을 남김. 대체로 날짜와 작업자 이름 기입
     2번 : 버전 설명에는 배포한 태그명을 작성. 또는 편의대로 설명을 추가해도 무방
     3번 : [게시]를 클릭하면 모든 과정이 완료되고 가상 태그가 웹 사이트에 구현
  • Process
    08
    태그 배포  
     "듣는 사람이 없는 것 처럼 노래를 부르세요" 라는 문구가 등장하면 태그가 올바르게 배포되고 있는 것
    태그 배포  
     배포가 완료되었으며 배포된 버전에 대한 설명 화면이 노출 됨
  • Process
    09
    태그 배포

    태그 배포  
     배포가 완료되었으며 배포된 태그는 태그 탭에 목록형으로 노출
